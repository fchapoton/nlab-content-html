Defunctionalization

Defunctionalization

Idea

Defunctionalization is a method of converting a functional program into one that involves no higher order functions nor lambda expressions. It is a whole program transformation.

Program transformation

The idea is it to consider a type Defun(A,B)Defun(A,B) for every pair of types A,B as a proxy for the true function type A→BA\to B. This type Defun(A,B)Defun(A,B) is a coproduct

Defun(A,B)=∐ x 1:C 1,…,x n:C n⊢λy.t:A→BC 1×…×C n Defun(A,B)=\coprod_{x_1:C_1,\dots,x_n:C_n\, \vdash \,\lambda y.\,t\, :\, A\to B}C_1\times \dots \times C_n

over all lambda abstractions λy.t\lambda y.\,t that appear syntactically in the program. Notice that lambda abstractions may have free variables x 1:C 1,…,x n:C nx_1:C_1,\dots,x_n:C_n, because they may appear deep inside the program. An inhabitant of the type Defun(A,B)Defun(A,B) is sometimes called a closure, because it is a syntactic expression together with an environment: a valuation for its free variables.

The type Defun(A,B)Defun(A,B) has a canonical evaluation function eval:Defun(A,B)×A→Beval:Defun(A,B)\times A\to B roughly given by

eval(((x 1:C 1…x n:C n⊢λy.t:A→B),(x 1,…x n)),y)=t eval(((x_1:C_1\dots x_n:C_n\vdash \lambda y.\,t:A\to B),(x_1,\dots x_n)),y)=t

and so it behaves like a function space.

Since AA, BB and the C iC_i‘s can also be defunctionalized, this is a method to transform a whole functional program into one with no lambda expressions nor any true function types.

The coproduct type Defun(A,B)Defun(A,B) is typically not a true function space in the sense of cartesian closed categories for two reasons.

  • it does not admit arbitrary lambda abstractions – just the lambda abstractions that happen to be included in the coproduct.

  • it does not equate lambda abstractions that are semantically equal but syntactically different.

(Thus it violates both existence and uniqueness in the universal property.)

Connection to adjoint functor theorems

Recall that a cartesian internal hom B AB^A, if it exists, is a representation of Hom(−×A,B)Hom(-\times A,B). As such, in many concrete situations, we can deduce its existence from an adjoint functor theorem. Recall that the solution set condition in this instance requires a set II and an II-indexed family of objects C iC_i together with morphisms

f i:C i×A→B f_i:C_i\times A\to B

such that every morphism f:C×A→Bf:C\times A\to B factors as

C×A→g×AC i×A→f iB C\times A \stackrel{g\times A}\to C_i\times A\stackrel{f_i}\to B

for some i∈Ii\in I and g:C→C ig\colon C\to C_i. Then, as is usually observed, the coproduct

∐ i∈IC i \coprod_{i\in I}C_i

is a weak internal hom, from which the internal hom is recovered by taking a coequalizer. This coproduct is reminiscent of the Defun(A,B)Defun(A,B).

References

Defunctionalization was probably introduced by John Reynolds in

  • John C. Reynolds, Definitional Interpreters for Higher-Order Programming Languages pdf

There and elsewhere a common theme is to first convert programs to continuation-passing style, which makes the control stack explicit, and then to defunctionalize, which makes the control stack look like a stack; the net result is a kind-of abstract machine.
